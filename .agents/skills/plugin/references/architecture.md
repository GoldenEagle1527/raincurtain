# 插件代码架构参考

本文档提供 Rain Curtain 插件开发的代码组织建议。**这不是强制模板**，而是为了应对复杂插件（如涉及大量 API 调用、复杂状态管理、多视图切换）时推荐的工程实践。

对于简单的工具型或展示型插件，将 HTML/CSS/JS 写在极少的文件中甚至是单文件中，完全是可以接受的。

---

## 一、代码组织建议（针对复杂插件）

当插件复杂度上升，代码量增加时，推荐按“职责分离（Separation of Concerns）”的原则来拆分代码，以提升可维护性。

### 1. 结构 (HTML)
*   **语义化：** 使用 `<header>`、`<main>`、`<nav>`、`<section>` 等语义标签，避免“div 地狱”。
*   **骨架与内容分离：** HTML 中尽量只放置页面静态骨架。列表、卡片等数据驱动的动态内容，推荐由 JS 动态生成并挂载。
*   **层级分区：** 推荐将主页面内容、弹窗（Modal/Overlay）、全局提示（Toast）分块放置，避免层级混乱。

### 2. 表现 (CSS)
*   **模块化命名：** 推荐使用 BEM (Block__Element--Modifier) 等清晰的命名规范，防止不同组件间的样式冲突。
*   **提取设计令牌：** 将常用的间距、响应式断点等提炼为 CSS 变量（如 `--spacing-md`）。
*   **使用主题系统：** 务必使用系统注入的 MD3 变量（`--md-primary`、`--md-surface` 等）来处理颜色，以完美适配深浅色主题。
*   **文件拆分：** 可以按基础样式（Base）、布局结构（Layout）、可复用组件（Components）、特定页面（Pages）来拆分 CSS，或简单地按组件拆分。

### 3. 行为 (JS)
*   **职责分离：** 避免将所有的网络请求、数据处理、DOM 操作混杂在一个巨大的函数或文件中。
*   **推荐的职责划分：**
    *   `api`：专注网络请求（`fetch`）和持久化存储（`RainCurtain.storage`）。
    *   `state`：专注数据的集中管理和状态变更逻辑。
    *   `ui`：专注 DOM 元素的创建、更新和事件绑定。
    *   `utils`：纯工具函数，不依赖业务状态或 DOM。
    *   `app` (或 `index`)：作为入口，协调上述各个模块的交互。

## 二、安全性与最佳实践

*   **DOM 操作安全：** 构建复杂的 DOM 时推荐使用 `document.createElement`。如果出于便利必须使用 `innerHTML`，请确保插入的数据经过了严格的 XSS 过滤或转义。
*   **作用域：** 尽量使用 IIFE（立即调用函数表达式）或 ES Modules，避免污染全局 `window` 命名空间。
*   **优雅降级：** 网络请求必须处理异常（`try...catch`），并在 UI 上给予用户友好的提示（例如加载失败、空状态）。
